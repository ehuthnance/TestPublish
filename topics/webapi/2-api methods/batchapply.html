Batchapply

The Batchapply API can be used to manage taxonomy, user, and product batch data on the server.

Method

Request URL

Description

POST

api/v7/BatchApply

Adds new batch data.

GET

api/v7/BatchApply?BatchCode={BatchCode}

Searches for an existing batch data using its batch code.

PUT

api/v7/BatchApply?BatchCode={BatchCode}&Status={Status}

Updates the specified batch data.

GET

api/v7/BatchApply?StartDate={StartDate}&EndDate}

Retrieves information on all the batch data that was created and modified within the specified time period.

HTTP request details

POST api/v7/BatchApply
Request information

URI parameters

None.

Body parameters

The following table describes the parameters available for the POST api/v7/BatchApply request.

Name

Description

Type

Required

batchID

Specifies the ID of the batch data. Length: 1 – 20.

Note: This parameter is optional. If the batch ID is not specified, the system automatically generates a batch ID.

string

No

applyDateTime

Specifies the date and time the batch data will be added.

date

Yes

fileCount

Specifies the total number of payload files to be added.

integer

Yes

fileType

Defines the type of the payload files to be added. The values for this parameter can be one of the following:

•product

•taxonomy

•user

string

Yes

fileFormat

Specifies the format of the control files to be added. The values for this parameter can be one of the following:

•application/json

•application/xml

string

Yes

payloadType

Defines the code representing the payload file type.

integer

No

filesList

Contains the list of data files and the order in which to load the files.

BatchApplyInformation

Yes

batchChecksumValidation

Enables or disables the checksum validation of the batch data. The value for this parameter can be one of the following:

•Yes

•No

string

No

autoStoreDataLoad

Flag indicating whether data should be loaded to the store automatically through the StoreDataLoad process. A value of zero, NULL, or greater than 2 indicates that there is no change in the existing process. The values for this parameter can be one of the following:

•NULL—send only updated data down to the affected store(s). This is the default behavior if the parameter is not included.

•0—send only updated data down to the affected store(s).

•1—push data for StoreDataLoad process after the batch is complete.

•2—no message published to queue, only insert data in database.

Note: This parameter and value are optional. When it is omitted or not specified, the software will behave as it did in previous releases.

integer

No

GET api/v7/BatchApply?BatchCode={BatchCode}
Request information

URI parameter

The following table describes the parameter forming the HTTP request URL.

Name

Description

Type

Required

Specifies the code of the batch data information to be searched.

string

Yes

Body parameters

None.

Response information

The following table describes the information retrieved by the GET api/v7/BatchApply?BatchCode={BatchCode} request.

Name

Description

batchCode

Specifies the batch code of the specified control file.

Note: The control file batch code must match with the batch code of the payload file.

tenantId

Specifies the ID of the tenant.

orgId

Defines the current taxonomy ID of the control file.

userName

Defines the name of the user who submitted the control file.

userCulture

Specifies the culture for the user who submitted the control file.

orgCulture

Specifies the culture for the organization associated with the user who submitted the batch data.

payloadData

Specifies the control file that contains the details for the BA processing.

applyStatus

Defines the status of the control file. The value for this parameter can be one of the following:

•0 (new)—the control file is new.

•1 (applying)—the control file is currently being applied.

•2 (success)—the control file has been successfully applied.

•3 (failed)—the control file has not been applied.

•4 (future)—the control file is set to be applied in a future date.

•5 (completed with error)—the control file has been successfully applied, but the application contains errors.

createDate

Specifies the date the batch data was created.

lastChangedTime

Specifies the date and time when the batch data was last modified.

fileDetails

Lists down the details of the control files in the batch data.

version

Specifies the version of the batch apply.

applyStartTime

Specifies the time when the batch apply for the batch data started.

applyCompleteTime

Specifies the time when the batch apply for batch data was completed.

totalNumberOfFiles

Defines the total number of control files in the batch data.

numberOfFilesPassed

Defines the number of control files in the batch data that were successfully applied to the server.

fileDetails

Defines the number of control files in the batch data that failed to be applied.

version

Specifies the version of the batch apply.

applyStartTime

Specifies the date when the batch data application was started.

applyCompleteTime

Specifies the date when the batch data application was completed.

totalNumberOfFiles

Specifies the total number of files in the batch data.

numberOfFilesPassed

Specifies the number of files in the batch data that were successfully applied.

numberOfFilesFailed

Specifies the number of files in the batch data that failed to be applied.

string

Status

Specifies the status of the batch data. The value for this parameter can be one of the following:

•0 (NEW)—the control file is new.

•1 (APPLYING)—the control file is currently being applied.

•2 (SUCCESS)—the control file has been successfully applied.

•3 (FAILED)—the application of the control file has failed.

•4 (FUTURE)—the control file is set to be applied in a future date.

•5 (CHECKSUM_VALIDATION_IN_PROCESS)—the control file has been applied, but contains errors.

•6 (PRE_CHECKING)—the control file has been applied, but contains errors.

•7 (IN_PROCESS)

•8 (PRE_CHECK_FAILED)
